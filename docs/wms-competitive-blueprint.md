# WMS Competitive Blueprint

This document benchmarks mature WMS products and translates proven patterns into the standalone WMS in this repository. It does not copy proprietary UI, wording, branding, or implementation details. It uses public documentation and product materials to identify common workflow patterns.

Reference sources:

- Microsoft Dynamics 365 Warehouse Management: work templates and location directives, mobile warehouse work (`https://learn.microsoft.com/en-us/dynamics365/supply-chain/warehousing/control-warehouse-location-directives`).
- Oracle WMS Cloud: receiving, putaway, replenishment, picking, cycle count triggers, tasking, barcode/RF flows (`https://docs.oracle.com/en/cloud/saas/warehouse-management/26a/owmim/implementation-and-configuration-guide.pdf`).
- NetSuite WMS: mobile receiving, putaway, inventory tracking, pick-pack-ship, cycle counting (`https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/section_156382533850.html`, `https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/section_N1229796.html`).
- Odoo Inventory/Barcode: SMB barcode operations, inventory adjustments, replenishment, practical routes (`https://www.odoo.com/documentation/18.0/applications/inventory_and_mrp/barcode.html`, `https://www.odoo.com/documentation/18.0/applications/inventory_and_mrp/inventory/warehouses_storage/replenishment.html`).
- SAP EWM-style operations: warehouse tasks, storage bins, internal movements, physical inventory (used as a conceptual benchmark alongside the public vendor docs above).
- Zoho Inventory: SMB bin locations, picklists, package/shipping flow, barcode scanning (`https://www.zoho.com/inventory/help/bin-locations/`).
- Cin7 Core/Omni WMS: mobile WMS dashboard, receiving, racking/putaway, picking, packing, stocktake, restock, stock transfer, labels (`https://help.core.cin7.com/hc/en-us/articles/9034461577487-Introduction-to-Warehouse-Management-System-WMS`, `https://help.omni.cin7.com/hc/en-us/articles/9128561211535-Overview-of-warehouse-management-WMS`).
- Fishbowl and ShipHero-style SMB/e-commerce WMS: mobile scanning, receiving, inventory control, pick-pack-ship, replenishment visibility (`https://www.fishbowlinventory.com/warehouse-management`, `https://www.shiphero.com/`, `https://software-help.shiphero.com/hc/en-us/categories/4419337429005-Picking-Packing`).

## 2. Best-Practice Synthesis

### Navigation

Serious WMS products separate daily work from setup. Workers start from tasks or operation flows, not warehouse tables. Managers get dashboards, exceptions, stock visibility, and audit/history. Admin setup is isolated.

Common workflow grouping:

- dashboard/task center;
- inbound: receiving and put-away;
- inventory: stock, movements, adjustments, counts;
- outbound: allocation, picking, packing, shipping handoff;
- replenishment;
- warehouse setup: warehouses, zones, locations, rules, labels;
- audit/reports/settings.

### Worker Guidance

Mature WMS worker UX uses guided steps:

1. scan or select the task;
2. scan location;
3. scan item;
4. confirm quantity;
5. show next action or completion.

Workers should see one primary action at a time. The system should reject wrong location/product scans immediately and say what to scan next.

### Status Tracking

Common status patterns:

- header status and line status are separate;
- worker task status is separate from inventory balance state;
- exceptions do not disappear; they become review/approval work;
- receiving and picking use partial states;
- cycle count has approval/rejection/recount states;
- packing/shipping handoff is separate from picking.

### Admin vs Daily Work

Setup screens should configure products, bins, rules, roles, and labels. Daily worker screens should not expose rule engines, enum values, or database objects.

### Exception Handling

Exceptions become visible manager work:

- short receipt;
- over receipt;
- damaged receipt;
- unknown barcode;
- wrong location/product scan;
- short pick;
- insufficient replenishment source;
- cycle count discrepancy;
- packing mismatch.

### Scanner Simplicity

Scanner flows should support keyboard wedge scanners:

- focused input;
- Enter advances;
- trimmed scan control characters;
- duplicate-submit protection;
- immediate validation;
- short Russian messages;
- mobile one-column layout.

## 4. Role-Based User Journeys

### Owner

- Sees first: `Обзор` with exceptions, open work, discrepancies, audit signals.
- Daily: checks bottlenecks, stock accuracy, users, reports.
- Hidden: worker-only scan forms unless entering an operation.
- Statuses: discrepancies, pending approvals, overdue/open tasks.
- One-click/simple: open journal, reconciliation, user settings.
- Requires approval: count approval, sensitive corrections, organization operations.

### Admin

- Sees first: `Настройки` or `Обзор`.
- Daily: user access, warehouses, locations, labels, setup.
- Hidden: worker task noise unless needed.
- Statuses: active/inactive setup objects, permission failures.
- One-click/simple: create user, assign role, configure location rules.
- Requires approval: no destructive ownership actions unless owner.

### Warehouse Manager

- Sees first: `Задачи` or `Обзор`.
- Daily: create/assign work, resolve exceptions, approve counts, manage products/locations.
- Hidden: organization ownership operations.
- Statuses: receiving discrepancy, short pick, recount required, replenishment needed.
- One-click/simple: generate put-away/replenishment/pick work.
- Requires approval: cycle count correction, manual stock corrections.

### Warehouse Worker

- Sees first: `Задачи`.
- Daily: receive, put away, transfer, count, pick, pack, replenish.
- Hidden: users, settings, products, barcodes, adjustments, approvals, journal.
- Statuses: assigned, in progress, completed, exception.
- One-click/simple: start next task, scan next barcode.
- Requires approval: adjustments, count approval, short pick resolution.

### Viewer

- Sees first: `Обзор` or `Товары и остатки`.
- Daily: reads stock, reports, audit when permitted.
- Hidden: all mutation actions.
- Statuses: stock availability, completed/open work, audit events.
- One-click/simple: search product/location, export/read reports.
- Requires approval: no actions.
